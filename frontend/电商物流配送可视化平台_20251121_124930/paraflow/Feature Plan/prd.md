# PRD — 电商物流配送可视化平台 MVP

## 1) 背景
当前电商物流场景中,商家和终端用户都面临着信息不透明、效率低下的问题。商家缺乏可视化工具来管理订单和规划配送范围,导致运营效率低、客户满意度差;用户在购买商品后只能看到模糊的物流状态,无法实时了解包裹位置,产生焦虑感。本次迭代旨在构建一个覆盖商家端、用户端和后端服务的全栈物流配送可视化平台,通过实时数据可视化技术,解决产品定位书中提出的"看不见、管不好、追不到"的核心问题,服务于商家李明的高效运营需求和用户王晓雨的安心期待场景。这是项目的首个完整迭代,将建立从订单创建到用户追踪的完整业务闭环。

## 2) 目标与期望成果
- 商家能够在5分钟内完成每日订单的筛选、查看和批量发货操作,将订单处理效率提升50%,减少手动操作带来的错误和延误
- 用户能够随时通过订单号查询包裹实时位置,在地图上看到包裹移动动画,获得准确的预计送达时间,消除等待焦虑,提升购物体验满意度
- 商家能够通过地图可视化工具圈选配送区域,根据配送时效筛选可送达订单,避免接到无法按时完成的订单,降低客户投诉率
- 平台提供稳定的实时数据推送能力,确保地图动画流畅、物流状态更新及时,建立用户对系统的信任感
- **非目标 / 边界:** 本次迭代不涉及库存管理、财务结算、多点路径优化算法、配送员调度管理等功能,专注于订单发货与物流追踪的核心闭环

## 3) 用户与故事
- **主要用户画像:** 商家 - 李明(高效运营者)
  - 故事 A: 作为商家,我希望在订单仪表盘上快速筛选和排序订单,这样我可以优先处理紧急订单,提高发货效率
  - 故事 B: 作为商家,我希望在地图上圈选配送区域并设置配送时效,这样我可以明确服务范围,避免接到无法按时送达的订单
  - 故事 C: 作为商家,我希望点击"发货"按钮后系统自动生成物流轨迹,这样我可以减少手动操作,让客户实时追踪包裹
  - 故事 D: 作为商家,我希望查看数据看板了解区域订单分布和配送时效统计,这样我可以优化配送策略,降低物流成本
- **次要用户画像:** 终端用户 - 王晓雨(安心期待者)
  - 故事 E: 作为用户,我希望输入订单号后立即在地图上看到包裹的当前位置,这样我可以知道包裹"在路上",不用担心丢失或延误
  - 故事 F: 作为用户,我希望看到包裹在地图上平滑移动的动画效果,这样我可以直观感受配送进度,获得心理安全感
  - 故事 G: 作为用户,我希望查看物流时间轴了解完整的配送历史,这样我可以清楚知道包裹经过了哪些节点,还有多久送达

## 4) 核心功能
- **商家订单管理仪表盘:** 提供订单列表展示、按状态筛选(待发货/运输中/已完成)、按时间和金额排序功能,支持点击订单查看详情,提供"发货"操作按钮触发后端轨迹生成,帮助商家高效管理大量订单
- **商家配送区域管理:** 提供地图拖拽圈选配送范围能力,自动识别区域内订单并显示可配送状态,支持设置不同物流公司的配送时效标准,根据时效筛选可送达订单,帮助商家精准匹配配送能力
- **用户物流实时追踪:** 提供订单号查询入口,在地图上展示包裹当前位置和预计送达时间,通过WebSocket接收后端推送的GPS坐标数据,实现车辆图标的平滑移动动画,让用户随时掌握包裹状态
- **物流时间轴展示:** 在地图下方或侧边以时间轴形式展示物流状态变更历史(已揽收/运输中/派送中/已签收),让用户了解完整的配送进度和关键节点信息
- **物流数据可视化看板 (加分项):** 提供区域订单密度热力图、配送时效统计图表、异常订单监控面板,使用ECharts或类似图表库实现多维度数据分析,帮助商家优化运营决策

## 5) 关键流程
- **场景:** 商家发货流程
  - **触发:** 商家登录系统后进入订单管理仪表盘,需要处理待发货订单
  - **路径:** 商家通过状态筛选器选择"待发货"订单,查看订单列表并点击某个订单查看详情(收货人信息、商品详情),确认无误后点击"发货"按钮;系统调用后端API将订单状态变更为"运输中",后端开始模拟生成该订单的配送轨迹数据
  - **结果:** 订单状态在仪表盘上实时更新为"运输中",商家收到发货成功提示,用户可以开始查询该订单的物流信息
- **场景:** 商家配送区域设置
  - **触发:** 商家需要明确配送范围,避免接到超出服务能力的订单
  - **路径:** 商家进入配送管理页面,在地图上拖拽圆形或多边形工具圈选配送区域;系统自动计算区域内的订单并显示可配送状态;商家选择物流公司并设置配送时效(如顺丰次日达、韵达3日达),系统根据时效筛选出可按时送达的订单列表
  - **结果:** 配送区域和时效规则保存成功,商家可以看到区域内可配送订单的数量和列表,后续发货时系统会自动校验订单是否在服务范围内
- **场景:** 用户查询物流
  - **触发:** 用户下单后希望了解包裹的实时位置和预计送达时间
  - **路径:** 用户打开物流查询页面,输入订单号点击查询;系统验证订单号后进入物流追踪页面,地图上展示包裹的当前位置(车辆图标)和预计送达时间;WebSocket连接建立后,后端每隔几秒推送最新GPS坐标,前端平滑移动车辆图标,模拟包裹在路上的状态;地图下方的时间轴同步显示物流历史状态节点
  - **结果:** 用户能够实时看到包裹位置和移动动画,获得准确的送达时间预估,可以随时刷新查看最新状态,消除等待焦虑
- **场景:** 商家查看数据看板 (加分项)
  - **触发:** 商家需要分析配送效率,找出问题区域并优化策略
  - **路径:** 商家进入数据看板页面,查看区域订单密度热力图(展示不同区域的订单分布),配送时效分析图表(统计平均配送时长的条形图或折线图),异常订单监控面板(实时显示配送超时或出现异常的订单);商家可以切换不同时间范围和维度查看数据
  - **结果:** 商家基于可视化数据发现配送瓶颈(如某区域订单多但时效差),调整快递公司选择或配送策略,优化整体运营效率

## 6) 竞品分析
- **竞争格局 (谁在解决这个问题):**
  - 菜鸟裹裹 - 面向C端用户的物流查询工具,聚合多家快递公司信息
  - 快递100 - 提供物流查询和管理服务,覆盖商家和用户
  - 京东物流 - 自营物流体系,提供实时追踪和配送管理
  - 顺丰速运APP - 单一快递公司的官方物流追踪应用
  - 传统ERP系统 - 企业内部订单管理系统,物流功能较弱
- **价值主张 (各方的承诺):**
  - 菜鸟裹裹承诺"一站式查询所有快递",但主要服务C端用户,商家管理功能薄弱
  - 快递100提供"商家和用户双端服务",但界面复杂,缺乏可视化能力
  - 京东物流强调"自营高品质配送",但仅限京东体系,不适用于多平台商家
  - 顺丰APP提供"精准时效承诺",但只能查询顺丰订单,无法聚合其他快递
  - 传统ERP系统侧重"订单和库存管理",物流追踪功能简陋,用户体验差
- **优势 / 劣势 (体验优缺点):**
  - 菜鸟裹裹和快递100的优势是聚合多家快递,但地图可视化能力弱,商家端功能不足
  - 京东物流的优势是实时性强、数据准确,但封闭体系限制了适用范围
  - 顺丰APP的优势是界面清晰、时效可靠,但单一快递公司局限性大
  - 传统ERP系统的劣势是学习成本高、界面陈旧,物流信息更新滞后
- **我们的差异化优势 (独特之处):**
  - 结合产品定位的"透明可视、高效智能、简洁优雅"关键词,我们提供商家和用户双端的深度可视化体验:商家通过地图圈选配送区域、实时查看订单分布热力图,用户通过地图动画"看见"包裹在路上,消除信息黑箱
  - 我们专注于电商物流场景的核心闭环,不做大而全的ERP系统,而是用实时数据推送和流畅的交互设计,让物流配送过程变得可感知、可预期、可信赖
  - 我们的技术栈(React + Node.js + WebSocket + 地图API)确保前端数据可视化的复杂度和实时交互的流畅性,这是传统物流系统难以匹敌的
- **切换成本与风险 (迁移成本和风险):**
  - 商家从传统ERP或Excel管理订单切换到我们的平台,需要适应新的界面和操作流程,但学习成本低,收益明显
  - 用户从菜鸟裹裹等工具切换过来,需要重新记住我们的查询入口,但地图可视化体验会成为吸引力
  - 主要风险是后端模拟轨迹数据的真实性和准确性,如果数据不可靠,会损害用户信任
